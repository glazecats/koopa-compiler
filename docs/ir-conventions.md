- Canonical IR is non-SSA, block-based, and verifier-backed; pass outputs must preserve entry reachability plus dense block/temp IDs.
- First-version lower IR is a downstream stage, not an in-place rewrite of canonical IR; phase 1 keeps the same function/basic-block/terminator CFG shape and only changes the value/storage boundary.
- First-version lower IR splits value refs from slot refs on purpose: value instructions consume temps/immediates only, while locals/globals appear only through explicit `load_*` / `store_*` slot operations.
- Lower IR must be able to accept any verifier-legal canonical IR, not only the output of the canonical default pass pipeline; canonical `ir_pass` cleanup may improve shapes, but it is not part of the lower-IR semantic contract.
- Current serial lowering preserves canonical CFG topology and canonical temp ids; any extra temps introduced to materialize slot reads are allocated above the canonical temp range.
- Current serial lowering rewrites canonical `mov` into either lower-IR `mov` or `store_*` depending on the destination kind, and rewrites canonical slot-valued `ret` / `br` conditions / `call` args / binary operands through explicit `load_*` instructions in the same source block.
- Lower-IR globals preserve canonical constant/runtime initializer flags as metadata, and the lower-IR verifier now mirrors the canonical runtime-init helper contract: reserved helper names are gated on runtime globals, `__global.init` / `__program.init` must keep the fixed zero-parameter zero-local single-block `ret 0` shape, and startup calls to `__global.init` are only legal from `main` entry or `__program.init` entry.
- Lower-IR verifier now also mirrors canonical top-level/public-call contracts: calls must resolve to known function entries with matching parameter counts, duplicate function/global names are rejected, and function/global name collisions are rejected.
- Lower-IR verifier must also be crash-proof on malformed metadata, not just semantically strict on well-formed tables: nonzero global/function/local/block counts with NULL backing tables are rejected explicitly rather than being allowed to reach raw table dereferences.
- Lower-IR verifier also treats count/capacity mismatches as structural contract failures, not “best effort” metadata: `global_count`, `function_count`, `local_count`, `block_count`, and `instruction_count` must all stay within their recorded capacities before any table walk begins.
- Lower-IR verifier now also treats CFG reachability as part of the public contract: body blocks must be reachable from `bb.0`, not merely locally well-formed and target-valid.
- Lower-IR builder APIs are no longer intentionally permissive on a few common malformed shapes: declaration-only functions fail fast on block append and temp allocation, parameter locals must remain a contiguous prefix, blocks reject instruction append after a terminator, and terminators are no longer silently overwritten.
- `lower_ir_lower_from_ir(...)` now self-verifies its own output before returning success, so lowering regressions fail at the production boundary rather than relying on every caller to remember an extra verifier pass.
- The current lower-IR phase-2 slice should now be treated as checkpoint-complete: representative lowering coverage, verifier hardening, and builder/API fail-fast are all in place, so further expansion should be need-driven rather than open-ended.
- The next likely structural step is value-SSA on top of lower IR, not more speculative hardening inside lower IR itself; use lower IR as the stable input layer and keep SSA scoped to value flow rather than memory SSA or machine/register concerns.

- Lower-IR verifier also now has a minimal declaration/signature shape contract: `parameter_count` must match the parameter-prefix locals, and ordinary declaration-only functions must not hide extra non-parameter locals or temp state. Reserved runtime-init helper names stay routed through their dedicated helper-shape rules instead of these generic declaration checks.
- Semantic return-path enforcement currently uses one explicit function-level boundary: a function definition is accepted for `SEMA-CF-001` only when statement-flow analysis proves both `guaranteed_return` and `!may_fallthrough` for the body summary. The loop-local `may_non_terminating` flag remains useful inside statement-flow heuristics, but it is not a second top-level veto once the body summary already proves a guaranteed return boundary; regressions now lock both sides of that policy for `while(1)` / `for(;;)` with partial-return versus all-path-return bodies.
- That same `SEMA-CF-001` loop policy is now also centralized instead of being open-coded twice: `while` and `for` return-flow decisions share one helper that applies the same guard-stability / loop-exit / body-return rules, and regressions now additionally lock nested infinite-loop all-path-return versus partial-return boundaries on both `while(1)` and `for(;;)` families, including the current mixed nested `for -> while` and `while -> for` shapes, so future edits do not drift between direct and nested loop shapes or between the two loop syntaxes.
- That same semantic loop policy now also has one first narrow backedge-proof slice beyond the older coarse "guard changed or not" heuristic: for constant-true loops with guarded partial exits, the current analysis may reject a body when it can explicitly prove the relevant exit guard is forced false again on the backedge (for example `while(1){ if(a) break; a=0; }`). This remains intentionally narrow and local rather than pretending to be full symbolic loop reasoning, but it closes at least the most obvious constant-false-backedge family without reopening broad false-positive risk.
- That same narrow backedge-proof slice is now also symmetry-locked across one extra mixed nesting boundary: the current `break`-guard refinement families are covered not only for direct `while(1)` / `for(;;)` loops and one-way mixed nesting, but also for the mirrored `while(1){ for(;;){...} return ...; }` and `for(;;){ while(1){...} return ...; }` shapes, so future helper edits do not quietly drift between the two mixed-loop directions.
- That same narrow backedge-proof slice now also reaches one first `for`-step family instead of stopping at loop bodies only: when a constant-true `for` loop's step expression itself provably re-forces the guarded partial-exit condition false before the next iteration (for example `for(;;a=0){ if(a) break; }`), the current semantic flow may now reject that loop too. This is still the same conservative local proof style, not a claim of full cross-iteration symbolic reasoning.
- That same local proof now also has one first tiny fallthrough-path refinement instead of treating `if(a) break;` as only “some exit may happen”. For direct identifier guards only, the continuing path may now reuse the obvious fact from that same branch split (`if(a) break;` implies `a==0` on the fallthrough path, and `if(a) ... else break;` implies `a!=0` on the continuing path). Current authority is intentionally narrow: this refinement is only there to support the existing backedge-proof family, not to promote the whole semantic flow engine into general path-sensitive symbolic execution.
- That same direct-identifier continuing-path refinement is now also regression-locked on both branch polarities, not only the `if(a) break;` side: constant-true `while(1)` and `for(;;)` loops now have focused accept/reject coverage for the mirrored `if(a) ... else break;` family too, so the helper cannot silently keep one polarity honest while drifting on the other.
- `machine_select` cleanup is no longer purely block-local plus "successor exists, so stay conservative". The current selected-side cleanup now has one first explicit CFG dataflow slice: block-entry must-agree alias facts for `copy` / `materialize_imm` may be propagated through the CFG to rewrite later op/terminator operands, and block `live_out` facts now decide whether a trivial def or pure value def is still needed past the current block before cross-block cleanup keeps or drops it. This is still intentionally smaller than a full late-machine optimization pipeline, but it replaces the older successor-peeking fallback with a real dataflow-backed selected cleanup boundary.
- That same selected-side dataflow line now also has one explicit `call` barrier boundary instead of assuming copy/materialize facts survive every call. Current cleanup invalidates caller-clobbered register aliases across selected `call*` ops while still allowing callee-preserved register facts to survive, and the focused `machine_select` regression matrix now locks both sides of that boundary.
- That same selected-side `call` barrier now also keeps spill-slot aliases honest: spill-backed values are still allowed to flow through the boundary and remain eligible for later fold-forward cleanup, so the current selected cleanup line distinguishes register-clobber pressure from stack-slot persistence instead of collapsing them into one generic “call kills everything” rule.
- That same selected-side alias propagation now also has explicit must-agree spill-slot join coverage in addition to the older register-side join cases: when mutually exclusive predecessors materialize the same spill-backed value, the join may now reuse that fact for later return/value cleanup rather than treating spill slots as CFG-propagation outsiders.
- That same spill-backed selected cleanup story now also covers the other two basic CFG sibling boundaries instead of only must-agree joins: unique-successor spill aliases may now survive into the successor for later operand/return cleanup, while disagreeing spill-slot predecessors stay intentionally uncollapsed so the join still preserves the explicit spill-backed value boundary instead of inventing a fake must-agree fact.
- That same selected-side live-out cleanup now also has explicit cross-block overwrite coverage instead of only same-block dead-def coverage: when a successor immediately redefines a register-backed or spill-backed resource before any surviving downstream use, the predecessor-side old def may now be dropped as non-live-out rather than being pinned just because control crosses a block boundary.
- That same live-out story now also has the matching partial-redefinition boundary locked on both resource families: if only some successor paths overwrite the resource but another path still carries the old value to a later use, the predecessor-side def remains live and must not be removed just because one sibling path happened to redefine it early.
- That same partial-path live-out boundary now also survives `call`-barrier pressure instead of being limited to plain redefinitions: a caller-clobbering call on one successor path does not let cleanup kill the predecessor-side register def when another path still carries that older value to a later use, while spill-backed values remain eligible for the stronger fold-forward path because the call barrier does not invalidate the spill resource itself.
- That same `call`-versus-live-out story now also has the complementary “all paths kill the old register, but not the old spill” boundary locked explicitly: if one successor path caller-clobbers a register and another successor path redefines it before the later use, the predecessor-side register def may be removed, whereas the analogous spill-backed def still stays live because the `call` path does not invalidate the spill resource.
- That same mixed-kill boundary is now also exercised in the more precise split form where different successor paths invalidate the same register by different mechanisms (`call` on one path, redefinition on another) while the analogous spill-backed resource remains live through the `call` path. Current authority is therefore no longer just “registers may die, spills may live” in the abstract, but “the live-out decision remains resource-kind-sensitive even when successor-side kill reasons differ across paths”.
- That same resource-kind sensitivity is now also exercised inside one predecessor block rather than only across one resource family at a time: under the same successor region, cleanup may drop a dead register-backed def while keeping a sibling spill-backed def live, and that surviving spill may then either remain explicit as `retspill` or be folded further to `reti` depending on the later path split, instead of forcing all predecessor defs to share one conservative keep/remove outcome.
- Lower-IR source-lowered regressions now also lock exact runtime-init dump shapes: `__global.init` writes runtime-initialized globals through `store_global`, `main` startup flow reloads globals through explicit `load_global`, and the no-`main` fallback preserves `__program.init` as the single-call wrapper around `__global.init`.
- Lower-IR exact-dump coverage now includes representative control-flow-heavy slot cases too: single-arm and two-arm `if` joins lower local writes through `store_local` plus join-block `load_local`, short-circuit branch conditions materialize per-block `load_local` values, and `while` backedges preserve canonical temps while inserting extra slot-materialization temps above the canonical temp range.
- Lower-IR exact-dump coverage now also reaches loop-step and multi-branch writeback shapes: `while`-break exits, `for` init/step CFGs, nested loop break/continue control, and branch-local `store_global` followed by join-block `load_global` are all regression-locked as current authority.
- Lower-IR exact-dump coverage also now includes call-heavy CFG slices: declared-call `for` conditions/steps (including short-circuit `&&` conditions) lower call args through explicit `load_local`, and branchy `main` functions with runtime-global startup still keep `__global.init` as the first entry instruction while later user-expression temps retain canonical numbering precedence.
- Lower-IR exact-dump coverage now also includes representative value-materialization shapes: local `&&` values, ternary value expressions, declared-call logical values, and nested logical-mix value materialization. At this point the current Phase 2 authority covers straight-line lowering, control-flow slot materialization, call-heavy CFGs, runtime-init/startup interaction, and value-materialization joins; further expansion should be need-driven rather than open-ended.
- CFG cleanup is currently limited to safe local rewrites: constant/same-target branch folding, empty-jump threading, `jmp -> ret`, and single-predecessor jump-block merge.
- Shared internal pass facts now include temp analysis (use counts, mov-copy facts, recursive temp constants), CFG analysis (reachable blocks, predecessor counts), and instruction side-effect classification.
- Safe fold extensions currently include immediate/immediate folding, single-immediate identity reductions, and same-operand simplifications; avoid folds that erase UB checks such as division/modulo by a possibly zero operand or invalid shift validation.
- Treat `call` as effectful even when its result temp is dead; direct DCE and the default pipeline must preserve those calls while removing surrounding pure work.
- Temp copy/constant facts are only sound for uniquely defined temps; verifier-legal cross-block join-temp materialization means shared temp analysis must invalidate those facts once a temp is defined on multiple mutually exclusive paths.
- Shared temp analysis now has explicit def/use-oriented facts too: definition counts and unique def-sites are first-class analysis outputs, and copy/constant propagation should key off those facts instead of hidden overwrite-prone state.
- Shared temp analysis now also records explicit unique use-site facts (including operand kind and site location), and temp-copy propagation is allowed to consume those facts for direct single-use rewrites before falling back to broader scans.

- Shared `IrValueRef` equality is now a reusable pass utility; CFG cleanup may use it for local value-preserving rewrites such as folding a branch whose two instruction-free successors both `ret` the same non-temp value.
- Same-return diamond folding is sound for equal temp returns too when both successor blocks are instruction-free: verifier-valid empty successors that `ret tmp.N` imply the temp is already available at the branch block exit, so the branch can collapse to `ret tmp.N` directly.
- CFG same-return diamond folding should compare returns by shared temp-analysis equivalence, not only by literal value-ref identity: safe constant resolution and safe copy resolution can expose equal returns even when the two empty successors use different temps.
- The same-return CFG rule now also covers successor blocks of the form `tmp = mov value; ret tmp` when both moved values resolve to the same branch-exit-available value; this stays conservative because the only erased instruction is the pure `mov`, and the folded `ret` uses the resolved source value rather than a successor-local temp.
- Dead-temp elimination must not treat every binary instruction as silently pure: dead-result `div`/`mod`/shift ops remain non-erasable because they still embody runtime validation / failure points even when their result temp is unused.
- Binary identity folding must not erase shift-by-zero operations under the current IR safety policy, because shift validity still depends on operand constraints beyond the immediate zero count.
- The default pass pipeline may need a post-DCE CFG revisit: temp propagation can canonicalize successor returns, DCE can then empty those successor blocks, and a final `simplify-cfg` + dead-temp cleanup pass can safely collapse the newly exposed branch diamond.
- Pipeline-level multi-def join-temp regressions should lock semantic path distinction, not a specific pre-cleanup join-block shape: full-pipeline CFG cleanup may legitimately sink the join `ret tmp.N` into the two successor return blocks as long as the branch and the distinct per-path return values remain preserved.
- Pipeline-level side-effect regressions should include CFG cleanup interaction, not only straight-line DCE: a dead-result `call` inside a branch that later becomes constant-folded must still survive after branch simplification, dead-block cleanup, and the post-DCE CFG revisit.
- For a few now-stable representative pipeline shapes, exact dump form is part of the regression contract, not just semantic equivalence: current examples include branch-local dead-result call cleanup, multi-def constant-path preservation, post-DCE same-return collapse to a single `ret` block, same-temp branch fold collapse to `ret 7`, and single-predecessor nonempty merge collapse to one surviving instruction plus `ret`.
- Representative pipeline regressions should also exercise the default pass order step-by-step, not only as one opaque pipeline call: verifier success after each individual pass and preservation of key invariants through the sequence are now part of the safety story.
- The public pass-pipeline contract now includes verifier gating at the API boundary: `ir_pass_run_pipeline(...)` rejects verifier-invalid input IR up front and rejects verifier-invalid pass output immediately after the offending pass.
- `memory_ssa` now also has a first real sibling skeleton: keep it under `include/memory_ssa.h`, `src/memory_ssa/`, and `tests/memory_ssa/`, separate from `value_ssa`. Current landed scope is still only the manual-construction/debug boundary: program/function/block/tracked-slot builders, version ids, block-entry memory phis, instruction-local memory access annotations, dump, verifier, and isolated regression/verifier coverage are now in place.
- `memory_ssa` now also has a first shared version def/use surface: `memory_ssa_compute_def_use_analysis(...)` materializes per-version defining slot ids, defining site kind/location (entry seed, phi, or instruction), use counts, and explicit use-site slices for phi inputs plus instruction memory uses.
- That shared Memory-SSA surface now also has first narrow consumer queries above it: `memory_ssa_resolve_load_store_value(...)` can recover a direct store-provided value when a load's reaching memory version is defined by a store instruction, `memory_ssa_resolve_store_previous_value(...)` can recover the overwritten prior value for a store when that incoming memory version is value-resolvable, and `memory_ssa_is_trivially_dead_store(...)` now also treats a store as dead when its defining memory version is used only as a later store's overwritten prior-version edge rather than by any observing load/phi/call use.
- `memory_ssa` now also has a first sibling pass module of its own: keep Memory-SSA-backed transforms under `include/memory_ssa_pass.h` and `src/memory_ssa_pass/` rather than threading Memory-SSA internals through the whole `value_ssa_pass` compilation unit.
- The first `memory_ssa_pass` slice is intentionally narrow but no longer local-only: `memory_ssa_pass_forward_local_loads(...)` and `memory_ssa_pass_forward_global_loads(...)` can forward loads from value-resolvable reaching memory versions, `memory_ssa_pass_eliminate_redundant_stores(...)` can erase same-value stores when the overwritten incoming memory version resolves to that same value, and `memory_ssa_pass_eliminate_dead_stores(...)` / `memory_ssa_pass_eliminate_dead_local_stores(...)` can erase zero-use stores from Memory-SSA use counts. These remain standalone passes for now, not default `value_ssa` pipeline stages yet.
- `memory_ssa_pass` now also has a first explicit mini-pipeline entrypoint: `memory_ssa_pass_run_pipeline(...)` currently sequences local/global forwarding, redundant-store cleanup, dead-store cleanup, a final forwarding revisit, and then a small existing Value-SSA cleanup tail (`simplify_trivial_values`, binary normalize/fold/identity cleanup, dead-value-def cleanup, and `simplify_cfg`). It verifier-checks its input up front, reruns that full sequence to a dump-stable fixed point, and rejects nonconvergence instead of being a one-shot best-effort helper. Treat it as a Memory-SSA-side experiment/canonicalizer, not as the default whole-compiler SSA pipeline yet.
- `memory_ssa_pass` now also has a first explicit memory-only canonicalization layer of its own: `memory_ssa_pass_canonicalize_memory_values(...)` runs the current Memory-SSA-backed load/store cleanup family plus the minimal Value-SSA cleanup needed to stabilize the resulting memory-value form (`simplify_trivial_values`, dead-value-def cleanup, and `simplify_cfg`) until it reaches a dump-stable fixed point. Prefer that API when a caller wants to normalize memory-derived SSA values without also pulling in the later arithmetic/value-only fold stages from the full pipeline.
- `memory_ssa_pass_eliminate_redundant_memory_binaries(...)` now also uses the shared memory-binary fixed-point budget consistently across its expose/join/full outer loops, so deeply nested straight-line scope shapes can finish their memory/CSE stabilization instead of stopping after the older hardcoded `8`-iteration ceiling.
- That Memory-SSA pipeline tail is now also strong enough to trigger both arithmetic cleanup and CFG-driven revisit opportunities that Memory-SSA itself merely exposes: cases such as `load_local a` forwarded to `7` and then `add ..., 5` can now finish as `ret 12`, while dead-path collapse after a forwarded branch condition can simplify the CFG and then feed another Memory-SSA iteration that was previously blocked behind a differing-value join or dead-path global call barrier.
- `memory_ssa_pass` source layout should now follow the same split-file discipline as the older pass layers: keep aggregator/plumbing in [memory_ssa_pass.c](/workspaces/compiler_lab/src/memory_ssa_pass/memory_ssa_pass.c), keep shared helpers in `memory_ssa_pass_core.inc`, and keep load-forward, store-cleanup, and pipeline logic in their own include fragments. The build now tracks those fragments explicitly through `MEMORY_SSA_PASS_SPLIT_INCLUDES`.

Current stage authority is best read as `M0/M1/M2` effectively complete enough for maintenance-first handling: representative slot-precise call modeling, scalar replacement, mixed-shape layering, and memory-aware cleanup ladders are now regression-locked across the current pure-global, mixed, materialized, partial-unknown, readonly-helper, and unknown-global-barrier families. Reopen this line for confirmed bugs, an explicit new downstream consumer need, or a deliberately chosen post-checkpoint Memory-SSA expansion, not for speculative broadening.
- Memory-SSA-backed forwarding and cleanup are no longer only join/diamond stories: the current authority now also covers representative same-value loop-header families. In particular, loop-carried memory phis whose incoming versions recursively resolve to the same value may now feed forwarded loads or redundant/dead store cleanup, while differing-value loop-carried phis still intentionally stop value resolution.
- Memory-SSA dead-store cleanup is now also no longer limited to straight overwrite chains: because `memory_ssa_is_trivially_dead_store(...)` can recurse through phi-result observation, stores whose only remaining influence is through a phi that itself only feeds later overwrites are now removable too. Current authority explicitly includes phi-overwritten and loop-overwritten local/global families.
- Memory-SSA value resolution is now slightly stronger than “direct defining store only”: `memory_ssa_resolve_load_store_value(...)` may also fold through a memory-phi when every incoming version recursively resolves to the same value, but it still intentionally bails out on entry-seeded versions, call-defined versions, cyclic self-dependence, or differing incoming values.
- Memory-SSA store accesses now also record their incoming/replaced memory version when one exists, not only their defining `mem.N`. The dump now exposes that as `mem.X = store_* slot @ mem.Y, value`, which gives later analysis/optimization code a first-class hook for “what value/state was this store overwriting?” rather than forcing it to reconstruct that edge indirectly.
- `memory_ssa_pass` now also has a first conservative redundant-store cleanup: `memory_ssa_pass_eliminate_redundant_stores(...)` removes a `store_local` or `store_global` only when the overwritten incoming memory version itself recursively resolves to the same value. That means same-value join cases can now collapse, while call barriers, entry seeds, cyclic memory, and differing-value phi joins still block the rewrite.
- Memory-SSA-backed load forwarding is now also stronger than “reaching version resolves to a concrete store value”: when two `load_local` / `load_global` instructions read the same memory version for the same slot and the earlier load dominates the later one, `memory_ssa_pass_forward_local_loads(...)` / `memory_ssa_pass_forward_global_loads(...)` may now rewrite the later load to `mov` of the earlier load result even if the underlying slot value itself is still unknown (for example parameter-local entry versions or post-call global versions).
- Memory-SSA-backed load forwarding is now also stronger than “one earlier dominating load exists”: for a join-block load whose reaching memory version is still unknown, if every predecessor already has some SSA value equivalent to reading that same slot from that same memory version, the forwarder may synthesize a small value-phi in the join block and rewrite the load to `mov` of that phi. Those predecessor-side values may be direct loads or trivial glue such as `mov(load)`, so unknown parameter/global values can stop reloading at joins without requiring every predecessor to keep the raw load result unwrapped.
- Memory-SSA-backed load forwarding is now also slightly PRE-like on acyclic joins: if a join-block load reads one memory version and some predecessors already materialize an equivalent value while other predecessors do not, the pass may conservatively insert missing predecessor-edge loads and then rewrite the join load to a phi of those predecessor values. Current authority intentionally excludes loop-header/backedge families; this is a narrow acyclic join slice only.
- Memory-SSA-backed load forwarding is now also stronger than “reuse a dominating raw load or synthesize a new join phi at the current load”: if some dominating scope already materializes an SSA value equivalent to reading the same slot from the same memory version, later loads may now reuse that existing equivalent value directly even when it is glue such as `mov(phi(load, load))` rather than a raw load instruction. This keeps later blocks from rebuilding redundant phis or reloads when the needed memory-derived SSA value is already live in scope.
- Memory-SSA now also has a first explicit local-slot promotion pass above raw load forwarding. `memory_ssa_pass_promote_local_slots(...)` rewrites `load_local` instructions to directly use a dominating value-phi when the corresponding memory version is itself defined by a dominating memory-phi whose incoming values can already be expressed in SSA. Current authority is intentionally narrow: this is local-slot-only, it promotes only when the memory-phi block dominates the load, and it does not try to invent a full scalar-replacement framework yet.
- That first local-slot promotion pass now also handles the memory-phi block itself, not only strictly later dominated blocks. When a join block's own `load_local` reads a local slot version defined by that same block's memory-phi, and the incoming memory versions already correspond to SSA-expressible values, `memory_ssa_pass_promote_local_slots(...)` may materialize a value-phi directly in the join block and rewrite the load to `mov` of that phi. This makes the first promotion slice useful on both join loads and some later dominated loads without claiming general scalar replacement.
- That first promotion slice now also has a global-slot sibling. `memory_ssa_pass_promote_global_slots(...)` applies the same dominating-memory-phi-to-value-phi rewrite to `load_global` when Memory-SSA proves the joined global version is already SSA-expressible, including same-value join/loop families, while call barriers that still leave the global version unknown continue to block promotion.
- Slot promotion is now also slightly stronger while assembling memory-phi inputs, not only at the promoted load site itself. When one phi input version does not resolve directly and the predecessor block itself does not materialize an equivalent SSA value, promotion may still reuse an already-materialized dominating ancestor value for that predecessor path. This covers mixed families such as "one predecessor writes back the entry-loaded value, the other just carries the entry version through" without forcing the predecessor block itself to reload or move the value first.
- Slot promotion now also avoids synthesizing vacuous value-phis when every promoted input is already the same SSA value or immediate. In those cases the promoted `load_*` is rewritten directly to `mov` of that common value instead of creating `phi [x, x, ...]` and leaving later cleanup to erase it.
- `memory_ssa_pass` now also has a first explicit higher-level local consumer above those individual rewrites. `memory_ssa_pass_scalar_replace_local_slots(...)` runs the current local-slot promotion, local load forwarding, local redundant/dead store cleanup, and a small SSA cleanup tail to turn simple local-slot carrier families into direct SSA value flow. Treat it as a deliberately narrow scalar-replacement experiment for locals, not yet a full slot-promotion framework.
- Redundant-store cleanup now also has a local-only entrypoint in addition to the whole-program one. `memory_ssa_pass_eliminate_redundant_local_stores(...)` applies the same “write back the value you already had” reasoning, but only to `store_local`, which makes it suitable as a building block inside local-slot-focused consumers without incidentally rewriting unrelated global stores.
- That higher-level consumer story now also has a global-slot sibling. `memory_ssa_pass_eliminate_redundant_global_stores(...)` and `memory_ssa_pass_eliminate_dead_global_stores(...)` expose global-only cleanup building blocks, and `memory_ssa_pass_scalar_replace_global_slots(...)` composes global promotion, global load forwarding, global redundant/dead store cleanup, and a small SSA cleanup tail into one narrow global-slot scalar-replacement experiment. Treat it as the global analogue of the local consumer, still conservative around call barriers and not yet a full memory-wide scalar-replacement framework.
- There is now also one combined public consumer above those local/global siblings. `memory_ssa_pass_scalar_replace_slots(...)` runs the local-slot and global-slot scalar-replacement consumers to a small fixed point, giving callers one explicit "erase simple slot-carrier memory flow into SSA values where possible" entrypoint instead of making them manually sequence the two halves.
- The first conservative memory-aware CSE slice now rides on top of that scalar-replacement layer rather than introducing a new parallel equivalence engine. Once slot-carrier loads/stores have been rewritten into stable SSA values/phis, the full Memory-SSA-backed pipeline may now hand the resulting arithmetic graph to `value_ssa_eliminate_redundant_binaries(...)`, which serves as the current memory-aware redundant-binary cleanup stage.
- That first memory-aware CSE slice is now also available as a direct public consumer instead of only being implicit inside the full pipeline. `memory_ssa_pass_eliminate_redundant_memory_binaries(...)` runs slot scalar replacement plus the existing Value-SSA binary normalization/fold/identity/CSE tail to remove repeated arithmetic over memory-derived SSA values without also pulling in the rest of the full memory-aware pipeline.
- That narrow memory-aware CSE stage is now also slightly more GVN-like than plain "same binary twice" elimination. It first exposes memory-derived SSA values without immediately dead-cleaning predecessor computations, then runs a join-local rewrite that can recognize a join-block binary over block-entry phis and replace it with a phi of per-edge results when those edge results are already recoverable either from predecessor-side binary computations or from per-edge immediate folding. This keeps the current scope deliberately narrow and join-local, but it is the first step beyond raw dominated-instruction CSE toward value-number-style reuse over memory-derived SSA expressions.
- That same join-local memory-aware CSE rule now also works per-edge rather than forcing one reuse mode for the whole join. After edge-local operand normalization, a rewritten join may mix predecessor-binary reuse on some incoming edges with per-edge immediate folding on others, and the full `memory_ssa_pass` pipeline now lets that rewrite run before later dead-value cleanup so predecessor-side arithmetic is still available to reuse. This is still deliberately narrow and join-local, not a general memory-aware GVN framework.
- That same join-local memory-aware CSE rule may now also project one more layer through same-block SSA glue in the join block itself. If a join-local binary feeds a later same-block `mov` or safe `binary`, predecessor projection can recurse through that local def and still recover per-edge results, so patterns like `sum = add(phi, phi); sum2 = add(sum, 5)` may now collapse to one phi of final edge results rather than stopping after the first joined binary.
- That same join-local memory-aware CSE rule now also has a first very narrow predecessor-materialization slice. When a safe joined binary can already be projected to one direct predecessor edge's operands but that predecessor does not yet contain the corresponding binary, the pass may materialize that binary at the end of the predecessor block and reuse it on the next fixed-point iteration. Current authority now includes both `jmp -> join` predecessors and branch predecessors whose `then` or `else` edge targets the join. This gives the current rule a small PRE-like capability without splitting CFG edges.
- That same join-local memory-aware CSE line now also projects/materializes recursively through one pure expression tree instead of relying on one fixed-point step per missing node. When a join-local value depends on earlier same-block `mov` / safe `binary` glue and some predecessor edges are missing several of those intermediate binaries, the current projection logic may synthesize the whole needed predecessor-side chain in one pass and still collapse the join to a final phi. This keeps the scope narrow to join-local pure expression trees, but it means deeper blocked-edge chains are no longer artificially limited by the old "one missing binary per iteration" behavior.
- Redundant-store cleanup is now also stronger than “prior memory version resolves to a concrete value”: `memory_ssa_pass_eliminate_redundant_stores(...)` may also erase a store when the stored SSA value is just a dominating `load_*` result (possibly through a `mov` chain) reading that same slot from that same incoming memory version. That means no-op stores like `ssa.0 = load_local a.0; store_local a.0, ssa.0` and `call touch(); ssa.0 = load_global g.0; store_global g.0, ssa.0` can now disappear even though the actual value remains unknown.
- That same load-origin tracking now also folds through phi glue more generally than “same source site”: if every phi input is itself equivalent to reading the same slot from the same incoming memory version, `memory_ssa_pass_eliminate_redundant_stores(...)` still treats the phi-fed store as “write back the value you already had”. This covers both same-source phis and join phis that merge sibling loads of the same underlying memory version.
- Memory-SSA call modeling is now also more precise than “every call kills every tracked global”. At this point the allocator move engine no longer has a broad "always recompute the whole value/family runtime state" step on incremental refresh; the remaining global recomputation is largely the full-reseed path itself.
- Spill-cost policy is now also one step richer than the original `uses + live_blocks + affinity` heuristic. Allocation prep now computes a natural-loop-based per-block loop-depth count from CFG dominators plus backedges, accumulates that into a per-value `loop_depth_sums` summary over the value's covered live blocks, and the current spill-cost total uses `use_count + (live_block_count + loop_depth_sum) + affinity_sum`. Straight-line cases keep the old totals, while values live through deeper loops are now less attractive spill candidates even before a fuller hotness model lands.
- Spill policy now also has a first narrow rematerialization-friendly tie-break. Allocation prep marks values defined by a direct `mov immediate` as `rematerializable`, and the spill-phase comparator now prefers spilling such values ahead of equally costly non-rematerializable values before falling through to effective degree and family move-work state. This is intentionally narrow: it does not yet change the printed spill-cost total, and it does not try to recognize arbitrary pure expression trees, but it means the allocator now has its first explicit "cheap to recompute later" hint in real spill choice.
- The optimistic select layer now treats select-time eviction as a real spill outcome, not only as "the blocker lost its color somehow". When reverse select evicts an already-colored lower-priority blocker, that blocker is now marked `spill-confirmed`, its stale preferred-color / targeted-eviction result metadata is cleared, and dumps/stats report it as a confirmed spill rather than a generic spill artifact.
- That same select-time eviction rule is now also less arbitrary on genuine ties. When multiple unique blockers are evictable at the same priority, the allocator now prefers blockers that were already `spill-intended`, then blockers marked `rematerializable`, then lower spill-cost blockers, before falling back to value-id order. This is still a local conservative heuristic, not a full optimistic-retry framework, but it makes late spill choice line up better with the allocator's earlier spill policy instead of degenerating to raw ID order.
- Reverse select now also has a first real optimistic-retry pass after the initial color assignment. Once the main reverse walk finishes, the allocator rescans currently spilled values and greedily recolors any one that has become colorable under the final colored neighborhood, for example because a later higher-priority value evicted one of its old blockers and the replacement does not interfere with it. Retry is still conservative: it does not perform fresh evictions, it just reuses now-free colors or preferred colors. On ties it recovers higher-priority, higher-cost, less-rematerializable, non-`spill-intended` values first.
- That retry path is now also first-class result state instead of an invisible late rewrite to the final color map. `ValueSsaAllocationResult` now records `spill_recovered_flags`, there is a public query helper for it, allocation dumps can mark colored values as `spill-recovered`, select traces distinguish `recovered-colored` / `optimistic-recovered` from ordinary `colored` / `optimistic-colored`, and aggregate select stats now report `spill_recovered_count`. That gives later optimistic-coloring work one more stable semantic bucket beyond only `spill-intended` and `spill-confirmed`.
- Retry is now also stronger than "only reuse a color that became fully free". When a previously spilled value is reconsidered after the main reverse-select walk, the allocator may now perform one more conservative late eviction if a color is blocked by exactly one lower-priority currently colored neighbor. Preferred-color retry still wins when available; otherwise retry may choose the best generic uniquely-blocking low-priority neighbor using the same spill-cheapness heuristics as the earlier select eviction logic. This is still bounded and deterministic: retry does not do arbitrary recoloring, only another unique-blocker step.
- Aggregate select stats are now also split enough to reflect that richer optimistic story directly. Besides `spill_recovered_count`, the allocator now reports `optimistic_direct_count` versus `optimistic_recovered_count`, so callers can tell apart values that survived `spill-remove` immediately from values that were only recovered by the later retry stage.
- That recovery surface is now also explicit about whether retry had to evict someone again to make the recovery happen. `ValueSsaAllocationResult` now records `used_retry_eviction_flags` plus `retry_eviction_blocker_value_ids`, there is a public query helper for that pair, select traces can print `retry-evict=ssa.N`, and aggregate select stats now report `retry_eviction_recovered_count`. So "recovered later" is no longer one opaque bucket; we can distinguish free-color recovery from recovery that still needed one more conservative unique-blocker step.
- The current result surface now also cleanly separates "ordinary targeted eviction during the main reverse-select walk" from "retry-stage eviction used to recover an earlier spill". Main-pass eviction still uses `used_targeted_eviction_flags` / `targeted_eviction_blocker_value_ids`; retry-stage eviction now has its own `used_retry_eviction_flags` / `retry_eviction_blocker_value_ids`. That avoids conflating "this value defended a preferred color when first selected" with "this older spill candidate only came back because retry later evicted a blocker for it."
- Before spilling a uniquely-blocking colored neighbor, the allocator now also tries one bounded recolor step on that blocker. If the blocker can move to another legal color immediately, the allocator keeps it colored and reuses the freed color for the waiting value instead of turning the blocker into a confirmed spill and maybe recovering it later. This now applies both in the main reverse-select walk and in late retry. It is still strictly local, not a general recoloring search: one blocker, one alternate color, no recursive chase.
- That recolor-first rule now also means some older eviction-shaped regression families have changed character in a good way: cases that previously looked like "evict blocker, then maybe recover blocker later" may now stay entirely in the colored set if the blocker can be moved aside immediately. In other words, the current optimistic path is starting to reduce confirmed-spill creation itself, not only get better at repairing confirmed spills afterward.
- Main-select candidate choice now also starts to account for that recolorability when multiple unique low-priority blockers are available. On equal-priority blocker choices, the allocator now prefers the blocker whose color can be freed by an immediate recolor, rather than choosing purely by later spill cheapness tie-breaks and only discovering afterward that another blocker would have avoided a spill entirely. This is still a local heuristic, but it is the first place where "can I avoid creating any new spill here?" affects which blocker family gets chosen, not just what happens after the choice.
- Late retry candidate choice now follows that same direction too. Among equally urgent spilled values that both currently need a unique-blocker recovery step, retry now prefers the candidate whose blocker can itself be recolored away, rather than delaying that distinction until after the candidate has already won on value-local spill cost. That means the retry queue is beginning to optimize for "best net colored-set improvement" rather than only "largest individual spilled value first".
- Retry ordering now also gives real weight to preferred recovery sources instead of treating them as an afterthought once a candidate has already won by raw spill-local metrics. When two provisional spill candidates are otherwise close enough, recovery options backed by a stronger preferred source (`coalesce-direct` before `coalesce-class` before plain `affinity`) are now chosen ahead of generic recovery options. This is one small but real step toward merged-family-aware allocation pressure: the retry stage is starting to preserve coalesced-family structure deliberately, not only opportunistically.
- That family-aware retry ordering now also looks at coalesced class size once the preferred-source tier itself ties. If two provisional spill candidates are both recoverable through the same kind of coalesced preference, retry now prefers the one whose recovery preserves the larger coalesced class instead of falling back immediately to value-local score or id order. This is still not merged-node allocation, but it is the first place where the allocator is explicitly valuing one coalesced family more highly than another during recovery choice.
- In other words, retry is now beginning to rank recovery options by a tiny notion of family value, not just value-local pain. The current ordering is still very small and local, but once two candidates reach the "same preferred-source tier" line, the allocator now treats preserving a larger coalesced family as intrinsically better than preserving a smaller one. That is the clearest current sign that post-select policy is bending toward merged-family allocation rather than purely scalar value choice.
- That same family-value idea now also exists one step earlier in the main select walk, not only in retry. When generic unique-blocker eviction must choose between otherwise similar blockers, main select now prefers evicting the blocker from the smaller coalesced family before it falls back to the older scalar tie-breakers. This is still a very local rule, but it means family-size preservation has crossed the boundary from "late recovery policy" into "initial spill-creation policy".
- The current main-select family rule is still intentionally narrow: it only engages after the earlier local gates such as lower priority and bounded recolorability are already satisfied. But once the allocator is in that "I must pick one blocker family" corner, it is no longer blind to family size. That makes the first spill-creation decision itself slightly more merged-aware instead of leaving all coalesced-family preservation to retry and recovery.
- Family value now also reaches the plan surface itself in one first narrow place. In spill-phase tie situations where cost, rematerialization, and earlier local gates have already tied, the allocator plan now prefers putting the smaller coalesced class into `remove=spill-remove` before it falls back to older scalar tie-breakers. So even before select runs, the spill-risk ordering has started to preserve larger coalesced families deliberately rather than only by accident.
- Family move-work state now also reaches the `spill` chooser, though still conservatively. After spill cost and active degree tie, the current planner prefers spilling `internal` families before `released`, `released` before `freeze-pending`, and `freeze-pending` before `coalesce-ready`, which means "already disposable" values are chosen ahead of still-promising move families without disturbing the older cost/degree policy layers.
- Spill-risk planning now also preserves family pressure one step more systematically than only the earlier class-size tie-break. Once spill cost, rematerialization, and move-work class have tied, the planner now prefers spilling families with lower external affinity weight and fewer external move-neighbor families before it falls back to coalesce-class size and later scalar tie-breaks. In practice that means "lightly connected move families spill first" is now a regression-locked part of the planner contract, not only an accidental consequence of ids or later select/retry repair.
- That same family-pressure signal now also reaches main select's generic unique-blocker eviction choice, not only the earlier planner. When two low-priority blockers are both equally legal generic evictees and the earlier local gates have already tied, main select now prefers evicting the blocker from the lighter-pressure family first: weaker move-work class, then lower external affinity weight, then fewer external move-neighbor families, before it falls back to coalesced-class size and scalar spill-cost ties. So "protect the more externally entangled family" is now a regression-locked select-time policy too, not merely a pre-select spill-risk heuristic.
- Late retry now also consumes that family-pressure signal at the candidate-selection layer instead of falling straight from preferred-source/class tiers back to scalar ties. Once the earlier retry gates have tied, the allocator now prefers recovering the heavier-pressure family first: stronger move-work class, then higher external affinity weight, then more external move-neighbor families, before it falls back to scalar spill-cost/id order. That policy is now regression-locked on both same-tier free-color recovery and same-tier generic retry-eviction competition, so the retry stage protects the more entangled family deliberately even when no preferred-source or class-size distinction remains.
- Retry's blocker side is now also starting to behave like a first-class policy surface instead of an afterthought. When two same-tier retry-eviction candidates would otherwise fall back to scalar candidate-local ties, the allocator now prefers the recovery that sacrifices the lighter blocker family first rather than always letting candidate-local spill cost/id dominate. In other words, late retry is now beginning to optimize both sides of the trade: preserve the more entangled candidate family, and among candidate-local ties prefer to pay with the less entangled blocker family.
- The family-pressure surface consumed by main select and late retry now also includes current active family member count, not only move-work class and external pressure. When the earlier family-pressure lanes tie, main select now prefers spilling the blocker from the family with fewer still-active members, and late retry now prefers recovering the candidate from the family with more still-active members. So the allocator is beginning to treat "how much live family mass is still in play right now" as part of the same preservation heuristic, rather than leaving member count hidden only inside queue-priority math.
- The allocator's family-pressure policy is now also starting to share one actual comparison surface across layers instead of being recopied piecemeal. Planner spill ordering, main-select blocker-family preference, and late-retry candidate-family preference now all bottom out in the same ordered family-pressure lanes (`move_work_class`, external affinity weight, external neighbor count, active family member count), and the move engine's spill-phase family queue tie-break now also uses the spill-phase move-work ranking rather than the simplify-phase one. This is still not a full move-worklist allocator mainline, but it materially reduces the risk that adjacent allocator stages drift into subtly different ideas of which family is "lighter" or "heavier".
