- For a few now-stable representative pipeline shapes, exact dump form is part of the regression contract, not just semantic equivalence: current examples include branch-local dead-result call cleanup, multi-def constant-path preservation, post-DCE same-return collapse to a single `ret` block, same-temp branch fold collapse to `ret 7`, and single-predecessor nonempty merge collapse to one surviving instruction plus `ret`.
- Representative pipeline regressions should also exercise the default pass order step-by-step, not only as one opaque pipeline call: verifier success after each individual pass and preservation of key invariants through the sequence are now part of the safety story.
- The public pass-pipeline contract now includes verifier gating at the API boundary: `ir_pass_run_pipeline(...)` rejects verifier-invalid input IR up front and rejects verifier-invalid pass output immediately after the offending pass.
- `memory_ssa` now also has a first real sibling skeleton: keep it under `include/memory_ssa.h`, `src/memory_ssa/`, and `tests/memory_ssa/`, separate from `value_ssa`. Current landed scope is still only the manual-construction/debug boundary: program/function/block/tracked-slot builders, version ids, block-entry memory phis, instruction-local memory access annotations, dump, verifier, and isolated regression/verifier coverage are now in place.
- `memory_ssa` now also has a first shared version def/use surface: `memory_ssa_compute_def_use_analysis(...)` materializes per-version defining slot ids, defining site kind/location (entry seed, phi, or instruction), use counts, and explicit use-site slices for phi inputs plus instruction memory uses.
- That shared Memory-SSA surface now also has first narrow consumer queries above it: `memory_ssa_resolve_load_store_value(...)` can recover a direct store-provided value when a load's reaching memory version is defined by a store instruction, `memory_ssa_resolve_store_previous_value(...)` can recover the overwritten prior value for a store when that incoming memory version is value-resolvable, and `memory_ssa_is_trivially_dead_store(...)` now also treats a store as dead when its defining memory version is used only as a later store's overwritten prior-version edge rather than by any observing load/phi/call use.
- `memory_ssa` now also has a first sibling pass module of its own: keep Memory-SSA-backed transforms under `include/memory_ssa_pass.h` and `src/memory_ssa_pass/` rather than threading Memory-SSA internals through the whole `value_ssa_pass` compilation unit.
- The first `memory_ssa_pass` slice is intentionally narrow but no longer local-only: `memory_ssa_pass_forward_local_loads(...)` and `memory_ssa_pass_forward_global_loads(...)` can forward loads from value-resolvable reaching memory versions, `memory_ssa_pass_eliminate_redundant_stores(...)` can erase same-value stores when the overwritten incoming memory version resolves to that same value, and `memory_ssa_pass_eliminate_dead_stores(...)` / `memory_ssa_pass_eliminate_dead_local_stores(...)` can erase zero-use stores from Memory-SSA use counts. These remain standalone passes for now, not default `value_ssa` pipeline stages yet.
- `memory_ssa_pass` now also has a first explicit mini-pipeline entrypoint: `memory_ssa_pass_run_pipeline(...)` currently sequences local/global forwarding, redundant-store cleanup, dead-store cleanup, a final forwarding revisit, and then a small existing Value-SSA cleanup tail (`simplify_trivial_values`, binary normalize/fold/identity cleanup, dead-value-def cleanup, and `simplify_cfg`). It verifier-checks its input up front, reruns that full sequence to a dump-stable fixed point, and rejects nonconvergence instead of being a one-shot best-effort helper. Treat it as a Memory-SSA-side experiment/canonicalizer, not as the default whole-compiler SSA pipeline yet.
- `memory_ssa_pass` now also has a first explicit memory-only canonicalization layer of its own: `memory_ssa_pass_canonicalize_memory_values(...)` runs the current Memory-SSA-backed load/store cleanup family plus the minimal Value-SSA cleanup needed to stabilize the resulting memory-value form (`simplify_trivial_values`, dead-value-def cleanup, and `simplify_cfg`) until it reaches a dump-stable fixed point. Prefer that API when a caller wants to normalize memory-derived SSA values without also pulling in the later arithmetic/value-only fold stages from the full pipeline.
- That Memory-SSA pipeline tail is now also strong enough to trigger both arithmetic cleanup and CFG-driven revisit opportunities that Memory-SSA itself merely exposes: cases such as `load_local a` forwarded to `7` and then `add ..., 5` can now finish as `ret 12`, while dead-path collapse after a forwarded branch condition can simplify the CFG and then feed another Memory-SSA iteration that was previously blocked behind a differing-value join or dead-path global call barrier.
- `memory_ssa_pass` source layout should now follow the same split-file discipline as the older pass layers: keep aggregator/plumbing in [memory_ssa_pass.c](/workspaces/compiler_lab/src/memory_ssa_pass/memory_ssa_pass.c), keep shared helpers in `memory_ssa_pass_core.inc`, and keep load-forward, store-cleanup, and pipeline logic in their own include fragments. The build now tracks those fragments explicitly through `MEMORY_SSA_PASS_SPLIT_INCLUDES`.
- Memory-SSA-backed forwarding and cleanup are no longer only join/diamond stories: the current authority now also covers representative same-value loop-header families. In particular, loop-carried memory phis whose incoming versions recursively resolve to the same value may now feed forwarded loads or redundant/dead store cleanup, while differing-value loop-carried phis still intentionally stop value resolution.
- Memory-SSA dead-store cleanup is now also no longer limited to straight overwrite chains: because `memory_ssa_is_trivially_dead_store(...)` can recurse through phi-result observation, stores whose only remaining influence is through a phi that itself only feeds later overwrites are now removable too. Current authority explicitly includes phi-overwritten and loop-overwritten local/global families.
- Memory-SSA value resolution is now slightly stronger than “direct defining store only”: `memory_ssa_resolve_load_store_value(...)` may also fold through a memory-phi when every incoming version recursively resolves to the same value, but it still intentionally bails out on entry-seeded versions, call-defined versions, cyclic self-dependence, or differing incoming values.
- Memory-SSA store accesses now also record their incoming/replaced memory version when one exists, not only their defining `mem.N`. The dump now exposes that as `mem.X = store_* slot @ mem.Y, value`, which gives later analysis/optimization code a first-class hook for “what value/state was this store overwriting?” rather than forcing it to reconstruct that edge indirectly.
- `memory_ssa_pass` now also has a first conservative redundant-store cleanup: `memory_ssa_pass_eliminate_redundant_stores(...)` removes a `store_local` or `store_global` only when the overwritten incoming memory version itself recursively resolves to the same value. That means same-value join cases can now collapse, while call barriers, entry seeds, cyclic memory, and differing-value phi joins still block the rewrite.
- Memory-SSA-backed load forwarding is now also stronger than “reaching version resolves to a concrete store value”: when two `load_local` / `load_global` instructions read the same memory version for the same slot and the earlier load dominates the later one, `memory_ssa_pass_forward_local_loads(...)` / `memory_ssa_pass_forward_global_loads(...)` may now rewrite the later load to `mov` of the earlier load result even if the underlying slot value itself is still unknown (for example parameter-local entry versions or post-call global versions).
- Memory-SSA-backed load forwarding is now also stronger than “one earlier dominating load exists”: for a join-block load whose reaching memory version is still unknown, if every predecessor already has some SSA value equivalent to reading that same slot from that same memory version, the forwarder may synthesize a small value-phi in the join block and rewrite the load to `mov` of that phi. Those predecessor-side values may be direct loads or trivial glue such as `mov(load)`, so unknown parameter/global values can stop reloading at joins without requiring every predecessor to keep the raw load result unwrapped.
- Memory-SSA-backed load forwarding is now also slightly PRE-like on acyclic joins: if a join-block load reads one memory version and some predecessors already materialize an equivalent value while other predecessors do not, the pass may conservatively insert missing predecessor-edge loads and then rewrite the join load to a phi of those predecessor values. Current authority intentionally excludes loop-header/backedge families; this is a narrow acyclic join slice only.
- Memory-SSA-backed load forwarding is now also stronger than “reuse a dominating raw load or synthesize a new join phi at the current load”: if some dominating scope already materializes an SSA value equivalent to reading the same slot from the same memory version, later loads may now reuse that existing equivalent value directly even when it is glue such as `mov(phi(load, load))` rather than a raw load instruction. This keeps later blocks from rebuilding redundant phis or reloads when the needed memory-derived SSA value is already live in scope.
- Memory-SSA now also has a first explicit local-slot promotion pass above raw load forwarding. `memory_ssa_pass_promote_local_slots(...)` rewrites `load_local` instructions to directly use a dominating value-phi when the corresponding memory version is itself defined by a dominating memory-phi whose incoming values can already be expressed in SSA. Current authority is intentionally narrow: this is local-slot-only, it promotes only when the memory-phi block dominates the load, and it does not try to invent a full scalar-replacement framework yet.
- That first local-slot promotion pass now also handles the memory-phi block itself, not only strictly later dominated blocks. When a join block's own `load_local` reads a local slot version defined by that same block's memory-phi, and the incoming memory versions already correspond to SSA-expressible values, `memory_ssa_pass_promote_local_slots(...)` may materialize a value-phi directly in the join block and rewrite the load to `mov` of that phi. This makes the first promotion slice useful on both join loads and some later dominated loads without claiming general scalar replacement.
- That first promotion slice now also has a global-slot sibling. `memory_ssa_pass_promote_global_slots(...)` applies the same dominating-memory-phi-to-value-phi rewrite to `load_global` when Memory-SSA proves the joined global version is already SSA-expressible, including same-value join/loop families, while call barriers that still leave the global version unknown continue to block promotion.
- Slot promotion is now also slightly stronger while assembling memory-phi inputs, not only at the promoted load site itself. When one phi input version does not resolve directly and the predecessor block itself does not materialize an equivalent SSA value, promotion may still reuse an already-materialized dominating ancestor value for that predecessor path. This covers mixed families such as "one predecessor writes back the entry-loaded value, the other just carries the entry version through" without forcing the predecessor block itself to reload or move the value first.
- Slot promotion now also avoids synthesizing vacuous value-phis when every promoted input is already the same SSA value or immediate. In those cases the promoted `load_*` is rewritten directly to `mov` of that common value instead of creating `phi [x, x, ...]` and leaving later cleanup to erase it.
- `memory_ssa_pass` now also has a first explicit higher-level local consumer above those individual rewrites. `memory_ssa_pass_scalar_replace_local_slots(...)` runs the current local-slot promotion, local load forwarding, local redundant/dead store cleanup, and a small SSA cleanup tail to turn simple local-slot carrier families into direct SSA value flow. Treat it as a deliberately narrow scalar-replacement experiment for locals, not yet a full slot-promotion framework.
- Redundant-store cleanup now also has a local-only entrypoint in addition to the whole-program one. `memory_ssa_pass_eliminate_redundant_local_stores(...)` applies the same “write back the value you already had” reasoning, but only to `store_local`, which makes it suitable as a building block inside local-slot-focused consumers without incidentally rewriting unrelated global stores.
- That higher-level consumer story now also has a global-slot sibling. `memory_ssa_pass_eliminate_redundant_global_stores(...)` and `memory_ssa_pass_eliminate_dead_global_stores(...)` expose global-only cleanup building blocks, and `memory_ssa_pass_scalar_replace_global_slots(...)` composes global promotion, global load forwarding, global redundant/dead store cleanup, and a small SSA cleanup tail into one narrow global-slot scalar-replacement experiment. Treat it as the global analogue of the local consumer, still conservative around call barriers and not yet a full memory-wide scalar-replacement framework.
- There is now also one combined public consumer above those local/global siblings. `memory_ssa_pass_scalar_replace_slots(...)` runs the local-slot and global-slot scalar-replacement consumers to a small fixed point, giving callers one explicit "erase simple slot-carrier memory flow into SSA values where possible" entrypoint instead of making them manually sequence the two halves.
- The first conservative memory-aware CSE slice now rides on top of that scalar-replacement layer rather than introducing a new parallel equivalence engine. Once slot-carrier loads/stores have been rewritten into stable SSA values/phis, the full Memory-SSA-backed pipeline may now hand the resulting arithmetic graph to `value_ssa_eliminate_redundant_binaries(...)`, which serves as the current memory-aware redundant-binary cleanup stage.
- That first memory-aware CSE slice is now also available as a direct public consumer instead of only being implicit inside the full pipeline. `memory_ssa_pass_eliminate_redundant_memory_binaries(...)` runs slot scalar replacement plus the existing Value-SSA binary normalization/fold/identity/CSE tail to remove repeated arithmetic over memory-derived SSA values without also pulling in the rest of the full memory-aware pipeline.
- That narrow memory-aware CSE stage is now also slightly more GVN-like than plain "same binary twice" elimination. It first exposes memory-derived SSA values without immediately dead-cleaning predecessor computations, then runs a join-local rewrite that can recognize a join-block binary over block-entry phis and replace it with a phi of per-edge results when those edge results are already recoverable either from predecessor-side binary computations or from per-edge immediate folding. This keeps the current scope deliberately narrow and join-local, but it is the first step beyond raw dominated-instruction CSE toward value-number-style reuse over memory-derived SSA expressions.
- That same join-local memory-aware CSE rule now also works per-edge rather than forcing one reuse mode for the whole join. After edge-local operand normalization, a rewritten join may mix predecessor-binary reuse on some incoming edges with per-edge immediate folding on others, and the full `memory_ssa_pass` pipeline now lets that rewrite run before later dead-value cleanup so predecessor-side arithmetic is still available to reuse. This is still deliberately narrow and join-local, not a general memory-aware GVN framework.
- That same join-local memory-aware CSE rule may now also project one more layer through same-block SSA glue in the join block itself. If a join-local binary feeds a later same-block `mov` or safe `binary`, predecessor projection can recurse through that local def and still recover per-edge results, so patterns like `sum = add(phi, phi); sum2 = add(sum, 5)` may now collapse to one phi of final edge results rather than stopping after the first joined binary.
- That same join-local memory-aware CSE rule now also has a first very narrow predecessor-materialization slice. When a safe joined binary can already be projected to one direct predecessor edge's operands but that predecessor does not yet contain the corresponding binary, the pass may materialize that binary at the end of the predecessor block and reuse it on the next fixed-point iteration. Current authority now includes both `jmp -> join` predecessors and branch predecessors whose `then` or `else` edge targets the join. This gives the current rule a small PRE-like capability without splitting CFG edges.
- That same join-local memory-aware CSE line now also projects/materializes recursively through one pure expression tree instead of relying on one fixed-point step per missing node. When a join-local value depends on earlier same-block `mov` / safe `binary` glue and some predecessor edges are missing several of those intermediate binaries, the current projection logic may synthesize the whole needed predecessor-side chain in one pass and still collapse the join to a final phi. This keeps the scope narrow to join-local pure expression trees, but it means deeper blocked-edge chains are no longer artificially limited by the old "one missing binary per iteration" behavior.
- Redundant-store cleanup is now also stronger than “prior memory version resolves to a concrete value”: `memory_ssa_pass_eliminate_redundant_stores(...)` may also erase a store when the stored SSA value is just a dominating `load_*` result (possibly through a `mov` chain) reading that same slot from that same incoming memory version. That means no-op stores like `ssa.0 = load_local a.0; store_local a.0, ssa.0` and `call touch(); ssa.0 = load_global g.0; store_global g.0, ssa.0` can now disappear even though the actual value remains unknown.
- That same load-origin tracking now also folds through phi glue more generally than “same source site”: if every phi input is itself equivalent to reading the same slot from the same incoming memory version, `memory_ssa_pass_eliminate_redundant_stores(...)` still treats the phi-fed store as “write back the value you already had”. This covers both same-source phis and join phis that merge sibling loads of the same underlying memory version.
- Memory-SSA call modeling is now also more precise than “every call kills every tracked global”. The bridge computes a conservative transitive per-callee global read/write summary over internal `value_ssa` functions and uses that to annotate calls slot-by-slot: globals a callee never touches get no Memory-SSA access, globals it only reads get a use-only access, and globals it may write still get a fresh def-version kill. Declaration-only/unknown callees remain conservative all-global read/write barriers.
- Memory-SSA phi value resolution is now cycle-aware: when resolving a memory phi whose input traces back to the phi itself through a loop backedge, that self-referential input is skipped rather than treated as “unresolvable”. If all non-cycle inputs agree on the same value, the phi resolves to that value. This enables load forwarding, redundant store elimination, and dead store elimination to work correctly on loop-invariant slot values.
- Memory-SSA phi value resolution now also uses a two-pass iterative wrapper for indirect phi cycles (phiA → phiB → phiA). The first pass resolves what it can; any PHI-defined versions that failed are reset and re-resolved on a second pass, which can now see values cached from the first pass. This is a defensive measure that ensures correctness even when phi resolution order causes intermediate phis to be finalized before their cycle partners.
- Memory-SSA analysis now also exposes a public version-level value resolution API: `memory_ssa_resolve_version_value(...)` takes a version ID directly (not an instruction site) and returns the resolved value if one exists. This decouples value resolution from specific load/store sites and makes the resolver available to future consumers such as loop-invariant analysis, cross-block value queries, and custom pass heuristics without requiring them to locate a specific load instruction first.
- Memory-SSA analysis now also exposes public cross-layer query APIs for the SSA-value-to-memory-version equivalence relation. `memory_ssa_value_matches_slot_version(...)` answers "does this SSA value represent the value of slot S at memory version V?" by tracing through mov chains, load instructions, and value-phi/memory-phi structure. `memory_ssa_find_block_equivalent_value(...)` answers "is there an SSA value in block B that equals slot S at version V?" by scanning block instructions and phis in reverse. These were previously pass-private helpers in `memory_ssa_pass_core.inc`; they are now public analysis-layer APIs in `memory_ssa.h` / `memory_ssa_analysis.inc`, and the pass layer delegates to them.
- `memory_ssa_pass` now also has a first one-shot lower-IR consumer entrypoint above its individual passes: `memory_ssa_pass_build_canonicalized_from_lower_ir(...)` lowers directly to Value-SSA and then runs the Memory-SSA-backed pipeline before returning. Treat it as the memory-aware sibling of `value_ssa_build_canonicalized_from_lower_ir(...)` when a caller wants one public entry that already includes join/loop-capable memory cleanup rather than the older straight-chain-only Value-SSA canonicalization surface.
- That lower-IR one-shot surface now also comes in the same two tiers as the in-memory program APIs: `memory_ssa_pass_build_memory_canonicalized_from_lower_ir(...)` stops after Memory-SSA-backed memory-value normalization, while `memory_ssa_pass_build_canonicalized_from_lower_ir(...)` continues through the later arithmetic/value-only cleanup tail. Use the former when a consumer wants stable memory facts without also folding arithmetic; use the latter when the desired product is the strongest current canonical Value-SSA endpoint.
- `value_ssa_pass` is now also a direct public consumer of that lower-IR Memory-SSA bridge rather than leaving it stranded in the Memory-SSA module namespace. `value_ssa_build_memory_value_canonicalized_from_lower_ir(...)` and `value_ssa_build_memory_canonicalized_from_lower_ir(...)` now delegate to the corresponding Memory-SSA-backed lower-IR one-shot builders, so a Value-SSA-facing caller can choose among three explicit entry tiers: classic canonicalization only, memory-value canonicalization, or full memory-aware canonicalization.
- `value_ssa_pass` now also exposes one unified lower-IR canonicalization surface above those wrappers: `value_ssa_build_from_lower_ir_with_canonicalization(...)` selects among classic, memory-value, and full memory-aware lowering modes via `ValueSsaLowerIrCanonicalizeMode`. Treat that mode-based API as the preferred high-level caller surface when the choice is dynamic or policy-driven, and treat the three named wrapper entrypoints as explicit convenience shorthands for fixed call sites.
- `value_ssa_pass` now also has a first concrete default caller helper above that mode surface: `value_ssa_build_default_from_lower_ir(...)` currently means "use the repository's preferred lower-IR -> Value-SSA canonicalization policy", which is presently `VALUE_SSA_LOWER_IR_CANONICALIZE_MEMORY_FULL`. Prefer that default helper for ordinary production-style callers that do not need to choose policy explicitly; use the mode-based API only when the policy is itself part of the caller's contract.
